### 2.3 M2M 通信框架（trunk · BML 层）— 最大工作量

#### M2M 框架 V1（2026-02-02）

初始骨架（21 文件，+2,926 行），建立核心组件：

- `m2m_cfg` — JSON 配置管理（串口参数、设备类型）
- `m2m_comm` — 串口收发主循环 + 帧解析
- `m2m_comm_sndbuf` — 发送缓冲区管理
- `m2m_comm_dispatcher` — 命令码路由分发
- `m2m_mmi_type.h` — 完整命令码定义（363 行）

#### 服务端 `m2m_srv` 完整实现（2026-02-12，`trunk/bml/modules/m2m_srv/`）

- 完整的异步通信框架：串口收发 → 帧解析 → dispatcher 路由 → IPC 响应
- **m2m_basic**：MCU 基础信息获取（版本、硬件状态）；MCU 版本更新后自动写入 SPP 属性
- **m2m_can**：CAN 总线数据透传（命令码 `0x004x`）；实车板测 **5/5 全部通过** ✓
- **m2m_sensor**：传感器数据采集与订阅，修复回调不触发的两个根本问题
- **m2m_gps**：GPS 定位数据获取与解析，修复协议字段顺序实现错误
- **m2m_uart**：UART 串口数据透传；实车板测 **4/4 全部通过** ✓
- **m2m_trans**：通用透传通道（0xFFF0 通道），新增独立子模块
- m2m_tdr / m2m_firmware / m2m_mgmt 子模块框架
- 修复 `m2m_comm_snd/snd_v2` 中 heap-buffer-overflow 安全漏洞
- 注册各子模块 MCU 串口命令处理函数

#### 客户端 `m2m_clt`（2026-02-12/24，`trunk/bml/modules/m2m_clt/`）

- 新增 `yx_m2m_trans` 通用透传客户端模块
- 新增 `yx_m2m_can` / `yx_m2m_uart` 接口（含完整协议命令码注释）
- 修复异步回调链路全链路 Bug（IPC seq 映射、RSP topic 路由错误）
- 修复 `TRANS_DATA_IND` 消息被错误路由到 `_handle_response` 导致解析失败

#### MCU 模拟器 `mcu_simulator`（2026-02-12，`trunk/app/modules/mcu_simulator/`）

- 全新 APP 进程：无真实 MCU 时的集成测试替代方案（34 文件，+5,104 行）
- 支持心跳 `0x0001`、参数 `0x0002`、GPIO `0x0004`、CAN `0x004x`、GPS `0x0043~0x0049`、定时器 `0x0071` 等

#### 测试模块拆分（`trunk/app/modules/test/`）

- 将 `test_m2m.c` 拆分为 8 个子模块测试文件，补充 `reset_request` 接口测试

## 七、关键技术决策记录

| 决策 | 内容 |
|------|------|
| SAL 层架构 | fs/sys 基础模块下沉到 OSIA 层，不再通过 BSL 框架注册，减少跨层依赖 |
| M2M IPC 响应路由 | 服务端 cache 的 topic 必须用 RSP topic，不能复用 REQ topic（否则 msgbox handler 接收不到） |
| cmd_type → IPC seq 映射 | MCU 应答帧不携带 seq，必须用映射表（参考 m2m_sensor.c） |
| 帧构造统一接口 | 所有帧必须用 `m2m_comm_sndbuf_push()`，禁止手动拼帧 |
| 协议序列化规范 | 禁止直接发送 C 结构体（对齐 + 字节序问题），必须逐字节手动序列化为大端格式 |
| 日志后端迁移 | 从自研日志替换为 zlog，支持多 sink、按时间切文件，路径统一为 /mnt/sdisk/yx_log |
| GB32960 多链路 | 主备链路分离，独立 FSM 状态机管理各链路连接状态 |
| 透传通道隔离 | 透传命令通过专用 0xFFF0 通道（m2m_trans），不经过普通命令路由 |
| AI 工具链规范 | 代码生成前必须调用 bsl-api/osia-api skill 查询接口，禁止凭记忆编写 API 调用 |
